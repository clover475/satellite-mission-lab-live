# 第④课 · 芯片架构调度台 · 讲师指引（course4-teacher-guide.md）

本课不做真实芯片性能模拟器，全部评分是课堂规则模型，用于比较方案之间的相对取舍，不代表真实芯片的性能、功耗、价格或具体产品架构。CPU/DSP/FPGA/专用硬件加速器统一称为"四类计算资源或实现选择"：CPU/DSP/加速器可能集成在同一颗 SoC 内；FPGA 可能是外部器件，也存在内置 FPGA 的 SoC，但并非所有 SoC 都带 FPGA。

## 一、课堂使用顺序

1. 进入课程四默认显示"先完成纸笔判断，再由教师开始调度实验"，A/B 两组分数都是 0，不自动显示正确方案或运行评价。
2. 教师可选择直接进入快速闯关模式（游戏①任务闪配 → 游戏②约束突然变化），也可以跳过小游戏直接切到架构调度模式。
3. 架构调度模式：第一轮点"全部交给CPU"演示任务集中和瓶颈风险；第二轮学生按资源特点重新分工；第三轮教师切换三场景，学生只修改部分任务再比较变化。
4. A/B 两组各自完成一套分工后，点"比较两组"，先看七个分维度差异，再看综合评价（非最高级措辞）。
5. 教师随时可按 T 打开教师辅助（先弹隐私确认），Esc 关闭并复位；H 显示下一级提示；R 直接显示参考答案；N 换一题——全部只在课程四标签激活时生效。

## 二、游戏①任务闪配 · 全部题库（11 题）与计分规则

计分：选典型答案 2 分／选可接受替代方案且教师确认解释合理 1 分／用过一级提示后答对最多 1 分／用过二三级提示后答对 0 分／答错不扣分。核心结论：没有脱离场景的唯一最佳架构。

### 设备界面
- 典型答案：CPU（无可接受替代）
- 理由：设备界面主要是事件响应和分支逻辑，CPU 通用灵活最合适，不需要为界面单独设计固定电路。
- 什么条件改变后选择可能变化：如果界面需要处理超高帧率的图形渲染，可能会引入专门的图形加速模块，但这已超出本课堂讨论范围。
- 一级提示：先想：这项任务的计算量大不大？重复吗？
- 二级提示：设备界面主要是"响应用户操作、显示状态"，计算量不大，逻辑分支多。
- 三级提示：四类资源里最擅长处理分支逻辑和常变软件的是哪一类？
- 常见错误：把界面这种轻量、多分支的任务也交给 FPGA/专用硬件，白白增加开发难度。
- 20 秒讲稿：设备界面是分支逻辑为主的轻量任务，CPU 通用灵活最合适，不需要为它单独定制电路。
- 60 秒讲稿：设备界面处理的是用户操作、状态显示这类事件驱动的逻辑，计算量不大但分支很多，这正是 CPU 最擅长的场景——通用可编程、能灵活应对各种条件判断。把这种任务放进 FPGA 或专用硬件加速器意义不大：它们的优势是高速重复运算，而不是复杂分支逻辑，反而会白白增加开发成本和复杂度。
- 技术边界：这里的"CPU"是课堂对"负责通用控制逻辑的处理器"的统一说法，实际产品里可能是应用处理器或 SoC 内的某个核心。

### 协议栈与连接管理
- 典型答案：CPU；可接受替代：FPGA
- 理由：协议栈处理大量分支判断和状态机，CPU 最省心；只有连接数极多、需要硬件卸载包处理时才会考虑 FPGA。
- 什么条件改变后选择可能变化：如果连接数暴增、需要硬件级别的包处理卸载，可以把部分协议处理下放到 FPGA；但这会显著增加开发复杂度。
- 一级提示：协议栈本质上是"收到什么消息，做什么处理"的一长串状态机判断。
- 二级提示：状态机和分支判断多的任务，通常交给最灵活的那一类资源。
- 三级提示：典型答案是 CPU；只有在连接数极多、需要硬件包处理卸载时才考虑 FPGA。
- 常见错误：认为协议管理需要"高速并行"就交给 FPGA，其实协议栈的瓶颈通常不是运算量而是分支判断的复杂度。
- 20 秒讲稿：协议栈是状态机加分支判断，CPU 处理这类逻辑最省心，也最容易调试和维护。
- 60 秒讲稿：协议连接管理需要跟踪连接状态、解析消息类型、处理各种异常分支，这是一长串的条件判断而不是重复的数值运算。CPU 的通用可编程特性正好适合这种任务，而且协议逻辑经常需要打补丁修改，CPU 上改代码远比改 FPGA 逻辑或流片快。只有当连接数极多、需要硬件级别的包处理卸载时，才会考虑把部分工作下放到 FPGA，但那属于更复杂的系统设计，不是本课堂的默认选择。
- 技术边界：真实系统的协议栈可能分层运行在应用处理器和某些硬件卸载引擎上，这里只讨论课堂简化后的"谁来管理连接状态"这一层。

### 数字滤波
- 典型答案：DSP、FPGA；可接受替代：专用硬件加速器
- 理由：数字滤波是重复的乘加运算，DSP 和 FPGA 都为此类运算做了优化；算法定型、要量产时专用硬件也是合理选择。
- 什么条件改变后选择可能变化：算法还在调整时优先 DSP/FPGA，方便修改系数；一旦滤波器系数固定且要大规模量产，专用硬件优势会上升。
- 一级提示：滤波是"对每个采样点反复做乘法和加法"的运算。
- 二级提示：重复的数字信号运算，通常交给针对这类运算优化过的资源。
- 三级提示：典型答案是 DSP 或 FPGA；算法稳定后专用硬件也合理，但不是唯一答案。
- 常见错误：认为滤波只有唯一正确资源，或者把滤波交给 CPU 却不知道这会造成延迟和功耗问题。
- 20 秒讲稿：数字滤波是重复的乘加运算，DSP 和 FPGA 都为此做了优化，量产阶段专用硬件也是合理选择，不止一个正确答案。
- 60 秒讲稿：数字滤波的核心操作是对每个采样点反复执行乘法和累加，这类运算是 DSP 电路结构专门优化的对象，FPGA 也能通过并行逻辑高效实现。如果算法还在调整阶段，DSP/FPGA 便于修改系数；一旦滤波器结构定型且要大规模量产，专用硬件加速器能把这部分逻辑固化，进一步降低功耗和单台成本。用 CPU 做滤波也不是"不能用"，只是遇到高采样率时容易造成延迟升高、功耗上升。
- 技术边界："滤波"这里指数字域的采样点乘加运算，不代表所有滤波场景的性能都完全等价于本课堂给出的相对档位。

### FFT
- 典型答案：DSP、FPGA；可接受替代：专用硬件加速器
- 理由：FFT 是高度标准化的重复计算，DSP 的乘加单元和 FPGA 的并行结构都适合；算法长期不变时专用硬件效率更高。
- 什么条件改变后选择可能变化：原型阶段可能还在试不同变换长度，适合 DSP/FPGA；一旦变换长度固定且大批量生产，专用硬件更划算。
- 一级提示：FFT 的计算步骤是固定的标准算法，重复率很高。
- 二级提示：和数字滤波类似，重复的标准化运算适合专门优化过的资源。
- 三级提示：典型答案是 DSP 或 FPGA；专用硬件在量产且算法稳定时也合理。
- 常见错误：把 FFT 交给 CPU 硬算，导致延迟和功耗明显变差却没有意识到原因。
- 20 秒讲稿：FFT 是标准化的重复计算，DSP/FPGA 都适合，算法稳定后专用硬件效率更高，不是唯一答案。
- 60 秒讲稿：FFT（快速傅里叶变换）的计算流程是固定的标准算法，涉及大量重复的蝶形运算，这正是 DSP 专用乘加单元和 FPGA 并行逻辑结构的强项。如果算法参数（比如变换长度）还在实验阶段，用 DSP 或 FPGA 方便调整；一旦确定下来并要大规模量产，把 FFT 固化进专用硬件加速器能显著降低功耗和单台成本。用 CPU 完成 FFT 在原理上可行，但重复运算量大时会明显拖慢延迟、拉高功耗。
- 技术边界：这里的评价只针对课堂设定的相对适配度，不代表某具体型号 DSP/FPGA/ASIC 做 FFT 的真实运算速度对比。

### 解调
- 典型答案：DSP、FPGA；可接受替代：专用硬件加速器
- 理由：解调涉及连续的信号运算，科研阶段用 DSP/FPGA 方便调参；算法稳定且大规模量产时专用硬件更有优势。
- 什么条件改变后选择可能变化：数据量增大、实时性要求提高时优先 FPGA/专用硬件；算法定型且大批量生产时专用硬件优势更明显。
- 一级提示：解调是把接收到的模拟/数字信号还原成比特的连续运算过程。
- 二级提示：科研原型阶段更看重能不能快速改算法，大批量生产阶段更看重效率和成本。
- 三级提示：典型答案：科研原型用 DSP/FPGA；数据量大、实时性要求高用 FPGA/专用硬件；算法稳定量产用专用硬件更有优势。
- 常见错误：认为解调只能用一种资源，或者忽视了"科研阶段"和"量产阶段"应该给出不同建议。
- 20 秒讲稿：解调是连续的信号处理运算，科研阶段用 DSP/FPGA 方便调参，量产阶段专用硬件更有优势，答案会随场景变化。
- 60 秒讲稿：解调需要把接收到的信号连续地转换回比特流，运算量较大且需要一定的实时性。科研原型阶段，算法细节还会调整，DSP 或 FPGA 便于快速修改参数；当数据量增大、实时性要求提高时，FPGA 或专用硬件加速器能提供更稳定的高速处理；一旦算法稳定并进入大规模量产，把解调逻辑固化进专用硬件，能同时压低功耗和单台成本。这是一个会随场景变化推荐答案的典型任务，没有脱离场景的唯一最佳选择。
- 技术边界：课堂只讨论"这类信号处理适合哪类资源"，不代表任何具体调制方式的真实解调实现细节。

### FEC 纠错解码
- 典型答案：FPGA、专用硬件加速器；可接受替代：DSP
- 理由：FEC 纠错解码计算量大且结构规整，FPGA/专用硬件效率更高；数据量较小或算法还在调试时 DSP 也可以接受。
- 什么条件改变后选择可能变化：纠错算法还可能调整时，DSP/FPGA 更灵活；算法定型、追求效率和量产成本时专用硬件更合适。
- 一级提示：纠错解码通常涉及规整的迭代运算，重复率很高。
- 二级提示：这类运算量大、结构固定的处理，FPGA 或专用硬件通常效率更高。
- 三级提示：典型答案是 FPGA 或专用硬件；数据量较小、算法未定型时 DSP 也可以接受。
- 常见错误：把纠错解码交给 CPU 处理大数据量的场景，导致延迟明显上升。
- 20 秒讲稿：FEC 纠错解码计算量大且结构规整，FPGA 或专用硬件效率更高，数据量较小时 DSP 也可以接受。
- 60 秒讲稿：FEC（前向纠错）解码通常涉及规整的迭代运算，比如反复的校验和似然计算，运算量较大但结构固定，这正是 FPGA 并行逻辑和专用硬件加速器的强项，能在低延迟下处理大数据量。如果数据量较小、纠错算法还在验证调整阶段，用 DSP 处理也是可以接受的替代方案，只是效率会打折扣。选择应该结合数据量和算法成熟度综合判断。
- 技术边界：这里的相对适配度只是课堂教学简化，不代表某具体 FEC 算法（如 LDPC、Turbo 码等）在真实芯片上的性能对比。

### 波束：方向与策略计算
- 典型答案：CPU、DSP；可接受替代：FPGA
- 理由：波束方向和策略计算涉及较多判断和相对较低频率的更新，CPU/DSP 足够；如果这部分也要求极高更新速率，可以考虑 FPGA。
- 什么条件改变后选择可能变化：如果方向计算的更新速率大幅提高、逼近实时高速更新的量级，可以考虑挪去 FPGA；正常场景下 CPU/DSP 已经足够。
- 一级提示：先区分：这是"算出该往哪个方向调整"的策略计算，还是"实际把每个通道权重快速刷新"的高速执行？
- 二级提示：策略计算涉及判断和相对不那么频繁的数值运算。
- 三级提示：典型答案是 CPU 或 DSP；如果这部分更新速率也变得极高，可以考虑 FPGA。
- 常见错误：把"方向策略计算"和"多通道高速更新"混为一谈，误以为波束相关任务只能整体丢给 FPGA。
- 20 秒讲稿：波束方向和策略计算是判断加相对低频的数值运算，CPU 或 DSP 就够用，不需要一开始就上 FPGA。
- 60 秒讲稿：波束赋形任务其实包含两个层次：一层是"往哪个方向调整、用什么策略"的计算，这类计算涉及较多逻辑判断，更新频率相对没有那么高，CPU 或 DSP 足以胜任；另一层是"把计算结果实时下发到每个天线通道"的高速权重更新，这才是需要 FPGA 或专用硬件的部分。本课堂特意把这两层拆成两张独立任务卡，就是为了避免"波束就该整体丢给 FPGA"这种笼统印象。
- 技术边界：真实相控阵系统里，方向策略计算和权重更新之间的界线可能更复杂，这里只做课堂简化的两层拆分。

### 波束：多通道权重高速更新
- 典型答案：FPGA、专用硬件加速器（无可接受替代）
- 理由：多通道波束权重的高速更新需要大量并行运算和很低延迟，只有 FPGA 或专用硬件能跟上更新速率。
- 什么条件改变后选择可能变化：通道数进一步增加或更新速率进一步提高时，专用硬件的效率优势会更明显；算法仍需频繁调整时优先 FPGA。
- 一级提示："多通道"+"高速更新"，意味着同一时刻有大量并行的数值计算要在很短时间内完成。
- 二级提示：这种高并行、低延迟的重复运算，是四类资源里哪一类的看家本领？
- 三级提示：典型答案是 FPGA 或专用硬件加速器；CPU/DSP 通常跟不上这种更新速率。
- 常见错误：把高速多通道权重更新交给 CPU，结果更新跟不上，波束方向出现明显滞后。
- 20 秒讲稿：多通道波束权重的高速更新需要大量并行运算和很低延迟，只有 FPGA 或专用硬件能跟上这个速率。
- 60 秒讲稿：相控阵天线的每个通道都需要独立的相位/幅度权重，波束方向变化时，这些权重需要在很短时间内同时更新完毕，通道数越多、更新越频繁，对并行处理能力和延迟的要求就越高。FPGA 内部可以针对这种"多路并行、结构固定"的运算搭出专门的并行数据通路；专用硬件加速器则可以把这套逻辑进一步固化以追求更低功耗和更小体积。CPU 和 DSP 按顺序或有限并行处理这类任务，通常难以满足高速多通道更新的实时性要求。
- 技术边界：这里说的"跟不上"是课堂规则模型里的相对适配度差异，不代表某具体 CPU/DSP 型号完全无法实现多通道更新。

### 温度监控
- 典型答案：CPU（无可接受替代）
- 理由：温度监控通常只需要简单读数和阈值判断，CPU 或简单的低功耗管理模块就够用；本页没有单独 MCU 选项时放 CPU。
- 什么条件改变后选择可能变化：如果需要极低功耗的常驻监控（主处理器休眠时也要工作），实际产品通常会有独立的低功耗管理控制器负责，但本页不提供这个选项。
- 一级提示：温度监控需要做的事情是什么？读数、比较阈值、必要时报警或降频。
- 二级提示：这类任务计算量很小，不需要专门的高速运算资源。
- 三级提示：典型答案是 CPU；真实产品里也可能由独立的低功耗管理控制器负责。
- 常见错误：把温度监控这种极轻量任务也放进 FPGA 或专用硬件，浪费资源。
- 20 秒讲稿：温度监控只是简单读数和阈值判断，CPU 就够用；本页没有单独的低功耗管理控制器选项，所以典型答案放在 CPU。
- 60 秒讲稿：温度监控需要做的事情很简单：定期读取传感器数值、和阈值比较、超限时触发降频或告警，这是非常轻量的任务，不需要专门的高速运算资源，CPU 完全能够胜任。需要提醒的是，真实产品里，温度监控和电源管理经常由一个独立的低功耗管理控制器（有时候是电源管理芯片里的一部分）负责，这样主 CPU 甚至可以在这段时间休眠以省电；本课堂的四类资源选项里没有单独的"管理控制器"，所以把它归入 CPU 是课堂简化，不代表真实产品架构。
- 技术边界："本网页没有单独的 MCU/管理控制器选项"是课堂简化，真实产品里温度监控常由独立的低功耗控制器负责。

### Wi-Fi 管理
- 典型答案：CPU（无可接受替代）
- 理由：Wi-Fi 协议栈管理同样是分支逻辑为主的任务，CPU 通用灵活最合适。
- 什么条件改变后选择可能变化：如果需要处理极高吞吐量的数据转发，可能会引入专门的网络处理硬件，但这超出本课堂讨论范围。
- 一级提示：Wi-Fi 管理主要在做什么？建立连接、维护协议状态、处理各种网络事件。
- 二级提示：和协议栈管理类似，分支判断和状态机为主的任务适合最灵活的资源。
- 三级提示：典型答案是 CPU。
- 常见错误：把 Wi-Fi 管理和"数字滤波"这类信号运算任务混淆，误以为需要专门的信号处理资源。
- 20 秒讲稿：Wi-Fi 管理是分支逻辑为主的协议管理任务，和协议栈类似，CPU 通用灵活最合适。
- 60 秒讲稿：Wi-Fi 管理涉及建立连接、维护协议状态、响应各种网络事件，这和协议栈与连接管理是同一类任务——分支判断多、计算量相对不大，CPU 的通用可编程能力正好适合。不需要为这类管理逻辑单独设计固定电路或高速并行结构。
- 技术边界：这里的"Wi-Fi 管理"指协议状态管理这一层，不包括射频前端和物理层的信号处理。

### 数据搬运和缓存管理
- 典型答案：CPU（无可接受替代）
- 理由：本页只讨论"谁负责配置和管理数据搬运"，CPU 负责配置和调度是典型答案；真实系统里高速搬运通常由内存控制器/DMA 完成。
- 什么条件改变后选择可能变化：如果数据吞吐量进一步增大，实际产品会更依赖专门的硬件数据通路（如 DMA），但本课堂不展开这部分设计。
- 一级提示：先区分：是"决定怎么搬、配置搬运参数"，还是"真正执行高速搬运动作"？
- 二级提示：本页只讨论前者——谁来配置和管理，不展开谁执行真正的高速搬运。
- 三级提示：典型答案是 CPU 负责配置和管理；实际高速搬运通常由内存控制器/DMA/硬件数据通路完成。
- 常见错误：误以为选了 CPU 就代表"所有数据搬运都是 CPU 一个字节一个字节地亲自去搬"。
- 20 秒讲稿：这里只讨论谁负责配置和管理数据搬运，CPU 负责配置调度是典型答案；真正的高速搬运通常由内存控制器/DMA 完成，课堂不展开。
- 60 秒讲稿：数据搬运和缓存管理可以拆成两层：一层是"配置搬运参数、决定什么时候搬、搬到哪里"的管理逻辑，这一层适合 CPU 处理，因为它是分支判断为主的调度工作；另一层是"真正把大量数据从一个地方搬到另一个地方"的执行动作，这在真实系统里通常由专门的内存控制器、DMA（直接内存访问）控制器或硬件数据通路完成，几乎不占用 CPU 的运算时间。本课堂的资源选项里没有单独的 DMA 选项，所以把"数据搬运和缓存管理"这张任务卡归入 CPU，代表的是"配置和管理"这一层，不代表 CPU 亲自执行每一次数据搬运。
- 技术边界：课堂不展开 DMA/内存控制器的具体设计，只提醒"配置管理"和"实际执行搬运"是两回事。

## 三、游戏②约束突然变化 · 全部条件变化事件（8 个）

规则：先给出三场景之一的典型分工作为初始状态，随机出现一项条件变化，学生只能移动 1~2 张任务卡，提交后即得 1 分参与分并显示以下讲解。

### 算法还要频繁修改
- 推荐调整：把之前放进专用硬件的信号处理任务（滤波/FFT/解调/纠错）挪回 DSP 或 FPGA。
- 原方案风险：专用硬件一旦流片就很难改，算法频繁变动会导致反复投片，前期成本急剧上升。
- 为什么场景变化会改变架构：算法稳定性直接决定了"改起来贵不贵"——越不稳定的算法，越应该放在容易修改的资源上。
- 20 秒讲稿：算法还要频繁改，就应该把信号处理任务放回 DSP/FPGA，专用硬件改一次代价太大。
- 60 秒讲稿：当算法还处于频繁修改阶段时，把计算逻辑放进专用硬件加速器意味着每次算法调整都可能要重新设计电路甚至重新流片，这个代价远高于在 DSP 上改系数或在 FPGA 上重新烧写逻辑。这也是为什么科研原型阶段的典型分工会优先选择 DSP/FPGA——它们保留了修改的空间，而专用硬件是用"修改的自由度"换取"运行时的效率"，只有算法基本定型后才划算。

### 电池只剩 30%
- 推荐调整：优先把高功耗任务（尤其放在 FPGA 上的密集运算）挪去功耗更低的专用硬件，或者降低同时运行的任务数量。
- 原方案风险：继续维持高功耗方案会加速耗尽剩余电量，可能导致关键通信中断。
- 为什么场景变化会改变架构：电量紧张时，功耗这一维度的权重会临时上升，即使专用硬件灵活性差，换来的低功耗也更重要。
- 20 秒讲稿：电量只剩 30%，应该优先降功耗，把密集运算挪去更省电的专用硬件或减少并行任务。
- 60 秒讲稿：电池电量紧张会临时改变优先级排序：平时可能优先考虑开发速度或灵活性，但电量告急时，"能不能撑到任务结束"变成第一位的问题。这时应该优先把功耗较高的资源换成更省电的选项，即使专用硬件的灵活性较差，在这个紧急节点上功耗优势更重要。这也说明架构选择不是一次性决定，而要随运行状态动态权衡。

### 延迟要求提高
- 推荐调整：把关键链路上的任务从 CPU 挪去 DSP/FPGA/专用硬件，减少排队等待。
- 原方案风险：继续用 CPU 顺序处理关键任务，可能无法满足新的延迟指标，导致响应变慢甚至丢数据。
- 为什么场景变化会改变架构：延迟主要来自"资源是不是顺序处理、适不适合这类运算"，CPU 通用但不是低延迟专精，遇到高要求时容易成为瓶颈。
- 20 秒讲稿：延迟要求提高，就要把关键任务从 CPU 挪到 DSP/FPGA/专用硬件，减少排队等待。
- 60 秒讲稿：CPU 通常按程序顺序处理任务，遇到多个信号处理任务同时需要执行时容易排队等待，这在延迟要求不高的场景下问题不大，但一旦延迟指标收紧，排队时间就会变成明显的短板。DSP、FPGA、专用硬件加速器要么有专门的运算单元，要么支持大量并行处理，能够显著缩短关键路径上的处理时间。

### 准备生产十万台
- 推荐调整：把已经稳定的信号处理任务（滤波/FFT/解调/纠错、多通道波束更新）逐步挪去专用硬件，压低单台成本和功耗。
- 原方案风险：继续用 DSP/FPGA 量产十万台，单台成本和功耗都会明显高于专用硬件方案，十万台的差距会被放大很多倍。
- 为什么场景变化会改变架构：量产规模越大，前期投入被摊薄得越多，专用硬件"前期贵、单台省"的特性就越划算。
- 20 秒讲稿：准备量产十万台，应该把稳定的信号处理任务挪去专用硬件，靠规模摊薄前期成本换取更低单台成本和功耗。
- 60 秒讲稿：专用硬件加速器的前期开发成本很高，但一旦分摊到十万台的产量上，每台承担的前期成本就会大幅下降；同时专用硬件在功耗和运行效率上通常优于通用资源。所以进入量产阶段后，只要算法已经稳定，把信号处理这类重复性任务迁移到专用硬件，往往能同时压低单台成本和整体功耗，这是量产规模放大后架构选择应该调整的方向。

### 前期研发预算有限
- 推荐调整：避免同时引入多种专用硬件/FPGA 设计，优先用 CPU/DSP 这类开发成本较低的资源完成任务。
- 原方案风险：预算有限时仍然大量使用 FPGA/专用硬件，会导致开发阶段就超支，项目可能无法按时完成。
- 为什么场景变化会改变架构：前期开发成本和资源种类、专用硬件占比直接相关，预算紧张时应该优先控制这个维度。
- 20 秒讲稿：前期研发预算有限，就要少用 FPGA/专用硬件，优先用开发成本较低的 CPU/DSP 完成任务。
- 60 秒讲稿：前期开发成本不仅取决于任务本身，也取决于用了多少种不同的资源类型——每多引入一种 FPGA 或专用硬件设计，就多一份开发、验证和调试的工作量。当预算有限时，应该优先考虑用 CPU/DSP 这类开发门槛较低、工具链成熟的资源去覆盖尽可能多的任务，把 FPGA/专用硬件留给真正必须的少数关键任务。

### 单台成本必须下降
- 推荐调整：把量大且计算规整的任务（滤波/FFT/纠错/波束高速更新）迁移去专用硬件，同时保留少量必须灵活的任务在 CPU/DSP。
- 原方案风险：继续用通用性强但单价较高的资源方案，单台成本很难达到目标。
- 为什么场景变化会改变架构：单台成本主要由所选资源的量产单价决定，专用硬件在大规模场景下单价优势最明显。
- 20 秒讲稿：单台成本必须下降，就要把规整的重复计算任务迁移到专用硬件，用规模效应压低单价。
- 60 秒讲稿：当单台成本成为硬性指标时，最有效的手段是把计算量大、结构规整、算法已经稳定的任务迁移到专用硬件加速器——它们在大规模生产下单位成本最低。同时也不需要把所有任务都做成专用硬件，界面、协议管理这类分支逻辑多的任务留在 CPU 上通常更划算，因为专用硬件处理这类任务反而没有优势。

### 现场需快速升级算法
- 推荐调整：把需要频繁现场升级的任务放在 CPU/DSP（软件升级）或 FPGA（可远程重新烧写逻辑），避免放进专用硬件。
- 原方案风险：如果关键算法被固化进专用硬件，现场几乎无法升级，只能等下一代硬件。
- 为什么场景变化会改变架构："现场能不能改"取决于所用资源是否支持软件/固件级别的更新，专用硬件在这一点上几乎没有弹性。
- 20 秒讲稿：需要现场快速升级算法，就不能把它放进专用硬件，CPU/DSP 或可远程烧写的 FPGA 才行。
- 60 秒讲稿：如果某项任务预期需要在设备已经部署到现场之后还能快速升级算法，那么专用硬件加速器几乎是不能选的——它在制造完成后核心功能很难更改。CPU/DSP 上的算法通常以软件形式存在，可以远程推送更新；FPGA 的逻辑也可以通过重新烧写来更新，虽然比软件升级复杂，但仍然可行。这类需求应该优先排除专用硬件方案。

### 数据量突然增加
- 推荐调整：把首当其冲的数据处理任务（滤波/FFT/解调/纠错/数据搬运管理）从 CPU 挪去 DSP/FPGA/专用硬件。
- 原方案风险：继续用 CPU 处理暴增的数据量，会迅速成为瓶颈，出现明显的延迟升高甚至丢数据。
- 为什么场景变化会改变架构：数据量增大直接放大了"吞吐量"和"数据搬运压力"这两个维度的重要性，CPU 在这两项上通常不占优势。
- 20 秒讲稿：数据量突然增加，要把数据处理任务从 CPU 挪到 DSP/FPGA/专用硬件，避免吞吐量成为瓶颈。
- 60 秒讲稿：数据量增加会同时放大对吞吐量和数据搬运压力的要求。CPU 按顺序处理数据，遇到数据量暴增时很容易成为瓶颈，导致延迟升高甚至丢数据；DSP、FPGA、专用硬件加速器要么有专门优化的运算单元，要么能并行处理大量数据，更适合应对数据量突增的场景。同时也要留意数据搬运和缓存管理这类配置调度任务是否需要更频繁地介入。

## 五、七维评价指标定义

延迟（越低越好）／吞吐量（越高越好）／功耗（越低越好）／灵活性（越高越好）／前期开发成本（越低越好）／量产单台成本（越低越好）／数据搬运压力（低/中/高，描述性维度）。全部使用"很低/低/中/高/很高"等相对等级，不代表真实瓦数、毫秒或报价，只用于同一课堂规则模型内的相对比较。
